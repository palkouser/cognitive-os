# Context safety

Retrieved content is untrusted data even when its source is verified. It cannot change Controller
budgets, Tool Plane policy, approvals, verifier requirements, repository instructions, provider tool
definitions, or memory/claim authority.

Scope and sensitivity filters run before body hydration. Host configuration rejects network,
approximate, learned, provider-selected, and automatic-write paths. Hydrated bodies pass the existing
secret detector before rendering; rejected content is represented only by candidate ID hash and an
exclusion code. Trace and lifecycle events never contain source bodies.

The deterministic classifier labels policy-override language, secret requests, fake system text,
tool authorization, repository override, budget override, and long encoded payloads. A signal does
not interpret or execute content. Safe retained text is escaped and enclosed by trust-labelled data
boundaries. Material contradictions receive exact-lineage warnings rather than authoritative model
resolution.

Required context that is secret-bearing, over-sensitive, stale, forged, unprovenanced, oversized, or
otherwise invalid blocks provider execution. Optional invalid context is excluded with an explicit
reason. The required verifier bundle covers provenance, token budget, trust, sensitivity, required
coverage, trace, safety, and source snapshots.
